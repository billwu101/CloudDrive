2Harness 安全分層 & 關鍵決策

Harness 四層防線 + 安全網

層機制
① 工具 allowlistagent 只看得到白名單工具（harness 強制，非 prompt）
② 執行層裁決每次呼叫決定論檢查 ← 越獄也擋得住
③ 可逆寫入顯示計畫 → 確認才執行
④ 不可逆動作TOTP 2FA（帶外、綁定動作）
安全網：trash + 版本可回復、activity_log 可稽核

核心：模型提議、harness 裁決。安全機制全在 model 之外。

關鍵決策與理由

操作方式
直接 SDK — REST API 無命令字串，結構上無法夾帶危險指令
不用 MCP
只接一家 Drive，直接型別化工具最簡單、控制最緊；多家整合再上 MCP
不用 CLI
命令字串是唯一會被夾帶執行的介面，把關難
AI 範圍
只做自然語言/模糊層；確定性操作走純程式（省錢、快、穩）
統計/彙總
LLM 產資料、程式算結果 — 統計不寫成 skill；LLM 抽成固定 schema 存起來，計數加總交給程式 → 格式恆定、之後 0 LLM
2FA
TOTP 開放標準、免費、不依賴 Google；只卡不可逆動作
3AI 成本策略 · Workflow 分層 · 擴展性

成本：編譯期 vs 執行期

LLM 只在「沒見過的自然語言」第一次花一次，凍結成 spec/範本；制式步驟、重跑、執行期一律決定論。

更深一層：LLM 產資料、程式算結果。抽取一次成 schema 存起來，統計/彙總由程式跑 → 格式恆定、之後 0 LLM。

Workflow 三層

  • Skill — 能力（做一件事）
  • Workflow 引擎 — 編排：觸發/控制流/狀態/生命週期（純程式）
  • LLM 編譯器 — 只把人話→計畫，不負責執行

光靠 skill 不夠：觸發/選取/彙總都不是 skill。借 n8n 的骨，不要它的皮。

能服務多少人？

  • harness = 共用程式 + per-user 情境
  • 無狀態 app server，狀態進 DB
  • async I/O + 水平擴展

瓶頸不是 harness，是 LLM 成本 & Google 配額 → 每人預算 + 退避重試 + job 佇列。

4建議落地順序
階段內容為什麼先做
1 連接與直接操作Google OAuth + 直接打 Drive API 的瀏覽/搜尋/搬移（非 AI）隨時有可用產品，AI 出問題也不影響核心
2 Agent + Harness型別化 Drive 工具 + harness 把關 + 唯讀/可逆 skill先把安全骨架立起來再加危險動作
3 2FA + 不可逆動作TOTP 綁定、challenge 表、step-up 流程有了閘門才開放永久刪除/分享
4 Workflow 引擎宣告式 spec + 執行器 + 計畫確認 + 範本/快取自然語言自動化是亮點，最後做
5 語意搜尋 / 校正 Skillpgvector 長在現有 Postgres；OpenCV 校正當選用 Skill加值能力，可逐步插拔
